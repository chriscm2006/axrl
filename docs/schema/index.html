Introduction

Accessibility Reporting Language (AXRL) is a data format designed to describe accessibility tests for Web pages, Native apps and other software. AXRL is designed to be a flexible and extendable data format. AXRL is designed as a JSON format, although because it is based on semantic web technologies, it could also be expressed in other formats.

Example: A basic web page test

This example shows the test for example.com, where the test produced 2 results. One of the results is "failed", and the second is "passed".

{
  "@context": "https://axrl.org/context/web-v1.json",
  "@type": "WebPageTest",
  "testSubject": "http://example.com",
  "results": [
    {
      "@type": "TestFailure",
      "impact": "Critical",
      "description": "Web pages must have a title",
      "remediation": "Add a title to the web page",
      "helpUrl": "https://dequeuniversity.com/rules/axe/3.1/document-title",
      "node": {
        "@type": "DOMNode",
        "codeSnippet": "<html>...</html>",
        "selector": "html"
      }
    },
    {
      "@type": "TestSuccess",
      "helpUrl": "https://dequeuniversity.com/rules/axe/3.1/color-contrast",
      "node": {
        "@type": "DOMNode",
        "codeSnippet": "<h1>Welcome</h1>",
        "selector": "h1"
      }
    }
  ]
}

@type property

An object's @typeproperty defines what that object is. By adding a type to all objects, that object is explicitly defined, instead of having to infer the type from its properties, or from the property it is on. Types can easilly be extended to add properties as they are needed. For example, WebPageTestand AndroidViewTest are both extensions on theUserInterfaceTest type. This gives them a number of shared properties, as some unique properties.

Type Structure

  • TestResult: The result of a test.
    • TestSuccess: The result of successful a test.
    • TestSkip: The result of skipping a test.
    • TestFailure: The result of failed a test.
    • TestAbort: The result of a test that was not completed.
  • InterfaceTest: A single test of an interface in a particular state.
    • WebPageTest: A single test of a web page in a particular state.
  • TestSubject: The some interface that tests can be performed on
    • HTMLPage: An HTML page that tests can be performed on
  • ImpactType: An issue hinders the use of an interface to some degree
    • Critical: An issue that stops users from accessing information or functionality
    • Serious: An issue that can make functionality difficult to use, or content difficult to understand
    • Moderate: An issue that can make functionality frustrating to use
    • Minor: An issue that has little to no impact on users
